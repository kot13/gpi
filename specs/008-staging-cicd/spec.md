# Feature Specification: CI/CD на стейджинг из GitHub

**Feature Branch**: `008-staging-cicd`

**Created**: 2026-06-07

**Status**: Draft

**Input**: User description: "нужно настроить CI/CD из github на стейджинг. Стейджинг - VPS ubuntu 22 LTS, без docker. Установлены все зависимости."

> **Язык**: вся спецификация MUST быть на русском языке (конституция GPI, принцип I).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Автоматический деплой на стейджинг (Priority: P1)

Разработчик вливает изменения в основную ветку репозитория GitHub. Система автоматически собирает приложение, прогоняет обязательные проверки и выкатывает новую версию на стейджинг-сервер (VPS Ubuntu 22 LTS). После успешного деплоя стейджинг открывается по настроенному домену и отражает актуальное состояние кода.

**Why this priority**: Без автоматического деплоя команда вынуждена обновлять стейджинг вручную — это замедляет проверку фич заказчиком и QA и повышает риск человеческой ошибки.

**Independent Test**: Сделать небольшое изменение в репозитории, влить в основную ветку и убедиться, что через разумное время стейджинг показывает это изменение без ручного вмешательства на сервере.

**Acceptance Scenarios**:

1. **Given** в основной ветке появился новый коммит, **When** срабатывает пайплайн деплоя, **Then** на стейджинге запущена новая версия приложения с этим коммитом.
2. **Given** деплой завершился успешно, **When** пользователь открывает публичный URL стейджинга, **Then** сайт отвечает без ошибок (HTTP 200 на главной локали).
3. **Given** деплой завершился успешно, **When** пользователь открывает `/admin` на стейджинге, **Then** панель управления контентом доступна для авторизованного пользователя.

---

### User Story 2 - Проверки качества перед выкладкой (Priority: P2)

Перед выкладкой на стейджинг система выполняет автоматические проверки качества кода (сборка и тесты проекта). Если проверки не прошли, деплой на стейджинг не выполняется, а команда получает явное уведомление о причине сбоя.

**Why this priority**: Стейджинг должен отражать работоспособный код; выкладка сломанной сборки блокирует тестирование и создаёт ложное впечатление о состоянии продукта.

**Independent Test**: Намеренно влить коммит, ломающий сборку или тесты, и убедиться, что стейджинг остаётся на предыдущей рабочей версии, а в GitHub виден статус неуспешного пайплайна.

**Acceptance Scenarios**:

1. **Given** коммит ломает сборку приложения, **When** запускается пайплайн, **Then** деплой на стейджинг не выполняется и статус помечен как неуспешный.
2. **Given** коммит ломает автоматические тесты проекта, **When** запускается пайплайн, **Then** деплой на стейджинг не выполняется.
3. **Given** все проверки прошли успешно, **When** пайплайн завершается, **Then** деплой на стейджинг выполняется автоматически.

---

### User Story 3 - Ручной перезапуск деплоя (Priority: P3)

Ответственный член команды может вручную запустить деплой на стейджинг из интерфейса GitHub без нового коммита — например, после смены секретов окружения или для повторной выкладки последнего успешного коммита.

**Why this priority**: Не все изменения окружения сопровождаются коммитом в код; ручной триггер снижает зависимость от SSH-доступа к серверу.

**Independent Test**: Запустить деплой вручную из GitHub для текущего коммита основной ветки и убедиться, что стейджинг обновлён или перезапущен.

**Acceptance Scenarios**:

1. **Given** у пользователя есть права на запуск workflow в репозитории, **When** он инициирует ручной деплой, **Then** пайплайн выполняется для выбранной ветки/коммита.
2. **Given** ручной деплой завершился успешно, **When** проверяется версия на стейджинге, **Then** она соответствует задеployенному коммиту.

---

### User Story 4 - Безопасное управление секретами (Priority: P2)

Секреты стейджинга (ключи БД, URL сервера, токены Payload и др.) хранятся только в защищённом хранилище GitHub и на сервере, но не попадают в репозиторий, логи пайплайна и публичные артефакты.

**Why this priority**: Утечка секретов через CI/CD — критический риск безопасности для CMS и базы данных.

**Independent Test**: Просмотреть логи успешного деплоя и содержимое репозитория — секреты не должны быть видны в открытом виде.

**Acceptance Scenarios**:

1. **Given** настроен пайплайн деплоя, **When** просматриваются логи выполнения, **Then** значения секретов замаскированы или отсутствуют.
2. **Given** разработчик клонирует репозиторий, **When** он ищет production/staging секреты в истории коммитов, **Then** секреты не обнаружены.

---

### Edge Cases

- Что происходит, если стейджинг-сервер недоступен во время деплоя? Пайплайн завершается с ошибкой; предыдущая версия приложения продолжает работать (если процесс не был остановлен преждевременно).
- Что происходит при одновременном запуске двух деплоев? Второй деплой ожидает завершения первого или отменяется — одновременная выкладка на один сервер не допускается.
- Что происходит, если миграция/синхронизация схемы БД завершилась ошибкой? Деплой прерывается; команда получает уведомление; требуется ручное вмешательство.
- Что происходит, если сборка успешна, но перезапуск приложения на сервере не удался? Пайплайн помечается неуспешным; по возможности сохраняется работоспособность предыдущей версии.
- Что происходит при первом деплое на «чистый» стейджинг с уже установленными зависимостями? Приложение разворачивается, подключается к существующей PostgreSQL и становится доступным по URL стейджинга.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST автоматически запускать пайплайн деплоя на стейджинг при появлении нового коммита в основной ветке репозитория GitHub.
- **FR-002**: Система MUST выполнять сборку приложения и автоматические тесты проекта до начала выкладки на стейджинг.
- **FR-003**: Система MUST прерывать деплой, если сборка или обязательные тесты не прошли.
- **FR-004**: Система MUST выкладывать артефакт на стейджинг-сервер (VPS Ubuntu 22 LTS) без использования Docker-контейнеров.
- **FR-005**: Система MUST перезапускать production-процесс приложения на сервере после успешной выкладки так, чтобы новая версия обслуживала запросы.
- **FR-006**: Система MUST поддерживать ручной запуск деплоя на стейджинг из интерфейса GitHub.
- **FR-007**: Система MUST хранить секреты окружения стейджинга в защищённом хранилище GitHub; секреты MUST NOT попадать в репозиторий и открытые логи.
- **FR-008**: Система MUST применять конфигурацию окружения стейджинга (включая публичный URL домена) при сборке и запуске.
- **FR-009**: Система MUST синхронизировать схему базы данных на стейджинге в рамках деплоя.
- **FR-010**: Система MUST отображать в GitHub статус каждого запуска пайплайна (успех/неудача) с понятным описанием этапа сбоя.
- **FR-011**: Система MUST ограничивать одновременные деплои на один стейджинг-сервер (очередь или отмена параллельного запуска).
- **FR-012**: Система MUST обеспечивать доступность публичного сайта и `/admin` на стейджинге после успешного деплоя.

### Key Entities

- **Пайплайн (Pipeline Run)**: Единичный запуск CI/CD — триггер (push/ручной), коммит, статус этапов (проверки, деплой), время начала/окончания, исполнитель.
- **Стейджинг-сервер**: VPS Ubuntu 22 LTS с установленными Node.js 20+, PostgreSQL 16, процесс-менеджером приложения и каталогом деплоя; без Docker.
- **Секрет окружения**: Пара имя/значение для стейджинга (URL БД, ключи Payload, домен и т.д.); хранится в GitHub Secrets и/или в файле на сервере вне репозитория.
- **Артефакт деплоя**: Собранное приложение (сборка Next.js + зависимости), готовое к запуску на сервере.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: После вливания коммита в основную ветку стейджинг обновляется до новой версии без ручного SSH-доступа в 95% случаев успешных пайплайнов.
- **SC-002**: Полный цикл «коммит → доступный стейджинг» занимает не более 15 минут при типичном размере изменений.
- **SC-003**: 100% коммитов, ломающих сборку или тесты, не попадают на стейджинг автоматически.
- **SC-004**: Команда может определить причину неудачного деплоя по статусу и логам в GitHub без доступа к серверу в 90% случаев.
- **SC-005**: После успешного деплоя главная страница стейджинга и `/admin` отвечают без ошибок 5xx в течение 2 минут после завершения пайплайна.
- **SC-006**: Секреты окружения не обнаруживаются в публичных логах пайплайна и в истории репозитория при аудите после настройки.

## Assumptions

- **Триггер деплоя**: автоматический деплой при push в ветку `main`; ручной запуск доступен дополнительно.
- **Стейджинг-сервер** уже подготовлен: Ubuntu 22.04 LTS, установлены Node.js 20+, npm, PostgreSQL 16, процесс-менеджер (PM2 или systemd), reverse proxy (nginx) при необходимости; Docker не используется.
- **Доступ к серверу** для CI/CD — по SSH с ключом, хранящимся в GitHub Secrets; отдельный deploy-пользователь с ограниченными правами.
- **База данных** PostgreSQL уже создана на стейджинге; строка подключения передаётся через секреты.
- **Домен стейджинга** уже настроен или будет указан в секретах до первого деплоя (в т.ч. переменная публичного URL для SEO и OG).
- **Обязательные проверки CI**: сборка приложения, unit- и integration-тесты проекта; E2E на стейджинге — опционально в первой версии.
- **Production-деплой** (боевой VPS/CDN) — вне scope данной фичи; настраивается отдельно.
- **Откат (rollback)** в v1 — повторный деплой предыдущего коммита вручную через GitHub; автоматический rollback — улучшение следующей итерации.
- **Медиафайлы** на стейджинге хранятся локально на сервере или в уже настроенном хранилище; миграция медиа — вне scope.

## Dependencies

- Репозиторий проекта на GitHub с веткой `main` как источником деплоя.
- Доступ администратора репозитория для настройки Secrets и workflow.
- SSH-доступ и права sudo/deploy-пользователя на стейджинг-VPS.
- Рабочая PostgreSQL на стейджинге с credentials.
- Существующие npm-скрипты проекта: `build`, `start`, `db:push` (или migrate), тестовые скрипты.

## Out of Scope

- Настройка production-окружения и CDN для боевого домена.
- Docker-образы и оркестрация контейнеров.
- Мониторинг и алертинг (Prometheus, Grafana, Sentry) — отдельная фича.
- Автоматический rollback при ошибках health-check после деплоя.
- E2E-тесты Playwright как блокирующий gate CI (можно добавить позже).
- Установка системных зависимостей на VPS (считается выполненной до начала фичи).
